# 智慧农业低代码分析与辅助决策平台 PPT 文案

## 项目立意

智慧农业真正缺少的，不只是传感器和数据，而是把“环境感知、数据分析、业务判断、结果展示”串成闭环的工程能力。本项目聚焦温室与试验田场景，围绕温度、湿度、土壤湿度、光照、PM2.5 等核心指标，构建了一套集数据采集、低代码编排、分析预测、辅助决策和可视化大屏于一体的智能农业平台。我们的目标不是做一个静态展示页面，而是做一个可运行、可演示、可扩展、可沉淀的农业数字化原型系统。

## 时代命题

当前农业数字化建设正在从“设备联网”走向“数据驱动决策”，但大量应用仍停留在数据采集和简单展示阶段，存在三类共性问题：一是前端看板能显示数据，却缺少业务解释，管理者看到了数值，却不知道下一步该做什么；二是后端分析能力分散，告警、预测、建议往往分属不同模块，难以形成统一决策链路；三是系统开发门槛高，农业场景一旦变化，就需要重复开发新页面、新逻辑和新接口，导致试点快、复制难。本项目希望回答的时代命题，就是如何用低代码方式把农业环境监测从“看数”升级为“读数、懂数、用数、决策”。

## 实践调研

在具体场景上，本项目以智慧农业环境监测为落点，围绕温湿度、土壤湿度、光照、颗粒物等指标构建数据链路。后端通过 MQTT 监听设备上报主题，将数据写入 SQLite 中的 `devices`、`sensor_data`、`device_status` 等表；当前端需要展示时，再通过 Flask 接口输出农业总览、时间趋势、风险告警、决策建议、产量预测和气候趋势等结果。为了保证项目既适合比赛演示也适合后续扩展，系统还内置了示范数据和演示工程，即使现场没有真实设备在线，也能自动生成样例历史数据并完成完整展示。

从项目实现可以看到，我们不是抽象地谈“智能农业”，而是把调研结论落实为可执行链路。演示工作流会按顺序执行 `get_sensor_info`、`db_query`、`analytics_summary`、`abstract_data_model` 等节点，先读取最新环境采样，再查询历史趋势，再生成总览、告警、建议、气候、产量、决策和报告摘要，最后通过 `output` 节点暴露为项目端口。演示大屏则直接绑定这些端口，展示文本亮点、实时传感器卡片、现场抓拍图片、风险分布饼图、历史趋势折线图、环境抽象模型卡、气候预测卡、产量预测卡和辅助决策卡，形成从感知到决策的完整业务闭环。

## 价值实证

本项目的价值不在于“做了很多图表”，而在于已经能够把农业数据转化为可验证的业务结论。系统会基于近 24 小时和近 48 小时数据自动计算农业总览，包括设备在线状态、风险分数、告警数量、数据新鲜度和环境观察结论；会基于最新值与历史值识别土壤湿度过低、温度异常波动、空气颗粒物偏高等风险；会输出面向管理动作的建议，如补水、通风、病害巡检、设备巡检；还会进一步给出未来 6 小时与 24 小时的微气候趋势、作物产量指数、亩产估算和优先决策动作。

更进一步，项目不是把这些结果停留在后端 JSON 中，而是通过前端组件把分析结果转化成可读、可讲、可展示的成果页面。例如 `agri-summary` 组件能够自动解析分析摘要，提取结论、告警和建议；`agri-model`、`agri-climate`、`agri-yield`、`agri-decision` 组件能够直接消费统一的农业模型数据契约；大屏“运行生成网页”后，还会自动轮询农业接口、天气接口和摄像头抓拍接口，实现演示页的动态刷新。这意味着项目已经具备真实答辩、课程展示和场景验证的实证基础。

## 破局之道

本项目的破局点，在于用“低代码工作流 + 统一数据契约 + 行业化可视组件”重构传统农业信息系统的开发方式。系统整体链路可以概括为：MQTT 接入设备数据，SQLite 负责本地存储与时序管理，Flask 提供分析与项目服务接口，工作流编辑器把复杂逻辑封装为可拖拽节点，大屏编辑器把分析结果封装为可编排组件，最终形成一套既能跑起来、又能讲清楚、还能继续扩展的农业数字化方案。

前端层面，项目不是单页面，而是三类协同界面。首页负责统一入口、演示工程打开、最近项目管理和账号登录；工作流编辑器负责拖拽节点、配置变量、配置端口、运行流程、单步调试和自动保存；大屏编辑器负责拖拽组件、绑定工作流端口、配置页面背景、复制粘贴组件、导入导出 JSON，并可一键生成独立展示网页。后端层面，系统则提供农业总览、趋势时序、告警、建议、报告、气候预测、产量预测、决策引擎、抽象模型、高级预测、用户登录和项目存储等接口，真正支撑了前端的低代码化和智能化。

## 壁垒构筑

本项目的壁垒首先来自行业化抽象，而不是通用页面拼装。工作流节点并非只有通用的开始、循环、分支和打印，还内置了 `get_sensor_info`、`db_query`、`analytics_summary`、`abstract_data_model`、`advanced_prediction` 等农业分析节点，能够直接面向环境监测和决策辅助。大屏组件也不是普通文本和图表的堆叠，而是专门设计了 `agri-sensor`、`agri-summary`、`agri-model`、`agri-climate`、`agri-yield`、`agri-decision` 等农业专用组件，天然适配农业业务语义。

第二层壁垒来自工作流与大屏之间的“项目端口”机制。项目中的分析结果不是散落在代码里，而是先写入工作流变量，再通过 `output` 节点暴露成端口，最后由大屏组件按类型进行绑定。这让系统具备了明确的数据边界和组件复用能力，也让“分析逻辑”和“展示逻辑”可以解耦演进。第三层壁垒来自统一数据契约。`abstract_data_model` 节点会输出包含总览、气候、产量、决策等结构的 `screen_contract`，使多个农业卡片可以围绕同一模型结果协同渲染。第四层壁垒来自高级预测能力，系统能够基于历史数据生成预测曲线图、原始趋势图、特征重要性图、相关性热力图、异常检测图、模型对比 CSV 和决策报告 JSON，并将这些结果直接转化为大屏可消费的图片 URL 或图表 CSV。

## 能力矩阵

### 前端核心功能

| 模块 | 核心能力 | 项目中的具体体现 |
| --- | --- | --- |
| 首页入口 | 统一进入、演示启动、项目管理 | 首页可直接打开演示工作流和演示大屏，支持最近项目、登录注册、本地项目与数据库项目入口 |
| 工作流编辑器 | 低代码编排与调试 | 支持开始、顺序、循环、分支、打印、传感器读取、SQL 查询、分析摘要、抽象模型、高级预测、输出端口等节点拖拽配置 |
| 运行与调试 | 流程执行、日志回显、单步调试 | 支持运行工作流、查看日志、调试开始、单步执行、继续执行、停止调试 |
| 大屏编辑器 | 可视化编排与发布 | 支持文本、图片、柱状图、折线图、饼图、传感器卡、模型卡、气候卡、产量卡、决策卡、摘要卡、天气卡 |
| 页面生成 | 可运行展示页输出 | 可一键生成独立网页，并自动轮询农业接口、天气接口和摄像头抓拍接口 |
| 项目协同 | 本地与云端双存储 | 支持本地仓库保存、JSON 导入导出、数据库项目保存与下载、最近项目管理 |

### 后端核心功能

| 模块 | 核心能力 | 项目中的具体体现 |
| --- | --- | --- |
| 数据接入 | 设备数据采集与落库 | MQTT 监听设备主题，将温度、湿度、土壤湿度、光照、PM2.5 等数据写入 SQLite |
| 数据治理 | 本地数据库与样例数据 | 建立 `devices`、`sensor_data`、`device_status`、`users`、`user_projects` 等表，并在无设备时自动补足演示数据 |
| 业务分析 | 农业指标解释 | 提供总览、时序、告警、建议、报告等接口，输出风险分数、环境观察、告警条目和建议动作 |
| 预测建模 | 趋势与产量研判 | 提供微气候趋势预测、产量指数与亩产估算、决策优先级排序、抽象模型构建 |
| 高级预测 | 模型训练与可视化产物 | 基于重采样、滞后特征、滚动统计、随机森林与梯度提升回归生成图片、CSV 与 JSON 分析结果 |
| 平台服务 | 用户与项目服务 | 提供登录注册、会话保持、数据库项目保存、项目读取和下载等服务能力 |

## 未来图景

面向未来，本项目可以从“演示级闭环”继续演进为“生产级闭环”。一方面，可以扩展多设备、多地块、多温室的数据接入能力，形成跨区域农业监测看板；另一方面，可以把当前以监测和建议为主的链路继续向控制闭环延伸，对接灌溉、通风、补光等执行设备，形成真正的自动化决策联动。与此同时，还可以叠加病虫害图像识别、作物长势识别、气象预报融合、云端多用户协作等能力，让系统从单点监测平台升级为农业数字孪生与智能决策平台。

更重要的是，本项目已经打下了可扩展的底座。前端有工作流编辑器和大屏编辑器，后端有统一接口和统一数据契约，中间有项目端口机制和抽象模型机制，因此后续无论扩展果园、温室、水产养殖还是乡村治理场景，都不需要推倒重来，只需要在现有框架上增加行业节点、分析规则和可视组件即可。这使本项目既适合当前比赛展示，也具备后续落地和持续演进的现实潜力。

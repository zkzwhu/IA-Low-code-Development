# 工作流节点与大屏组件详细说明

本文档基于当前项目中的实际实现整理，目标不是只列出“有什么”，而是说明它们在整套系统里分别承担什么职责、如何衔接、应该怎样使用。

## 1. 先理解这套系统的四层结构

在当前项目里，工作流和大屏不是直接连在一起的，中间还有“变量”和“项目端口”两层。

### 1.1 四层概念

| 层级 | 名称 | 作用 |
| --- | --- | --- |
| 第 1 层 | 工作流节点 | 负责控制流程、获取数据、生成分析结果。 |
| 第 2 层 | 工作流变量 | 负责暂存节点计算出来的结果。 |
| 第 3 层 | 项目端口 | 负责把工作流变量暴露给大屏应用。 |
| 第 4 层 | 大屏组件 | 负责把端口数据渲染成文字、图表或专题卡片。 |

### 1.2 一条完整数据链路

```text
工作流节点
  -> 写入工作流变量
  -> output 节点暴露为项目端口
  -> 大屏组件绑定项目端口
  -> 最终展示在大屏页面上
```

### 1.3 为什么要经过 `output` 节点

很多人会自然地以为“节点写入了变量，大屏就应该能直接看到”，但当前项目不是这样设计的。

- 节点只是把结果写进工作流内部变量。
- 只有 `output` 节点会把变量正式暴露成“项目端口”。
- 大屏编辑器绑定的是“项目端口”，不是工作流内部变量本身。

所以，如果某个节点执行成功了，但大屏侧看不到数据，最常见的原因之一就是：

`结果已经写入变量了，但没有经过 output 节点暴露出来。`

## 2. 工作流节点详解

当前工作流节点大致可以分成三类：

1. 入口节点
2. 流程控制节点
3. 数据处理与对外输出节点

## 3. 所有节点都会涉及的几个关键字段

为了看懂节点配置，先把几个反复出现的字段说明清楚。

| 字段 | 常见出现位置 | 含义 |
| --- | --- | --- |
| `name` | 几乎所有节点 | 节点名称，用于编辑器展示、调试和识别。 |
| `nextNodeId` | 大部分顺序执行节点 | 当前节点完成后跳转到哪个节点。 |
| `variableId` | `print`、`output` | 表示“读取哪个变量”。 |
| `targetVariableId` | `get_sensor_info`、`db_query`、`analytics_summary`、`abstract_data_model` | 表示“把结果写入哪个变量”。 |
| `messageSource` | `print` | 决定打印的是固定文本还是变量值。 |
| `loopConditionType` | `loop` | 决定循环按次数执行还是按表达式执行。 |
| `branchCondition` | `branch` | 分支条件的占位结果，用来控制走真分支还是假分支。 |

这里最容易混淆的是：

- `variableId` 更偏向“读取变量”
- `targetVariableId` 更偏向“写入变量”

## 4. 入口与流程控制节点

### 4.1 `start` 开始节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 整个工作流的执行入口。 |
| 核心配置 | `nextNodeId` |
| 适合场景 | 所有工作流都必须从这里开始。 |

详细说明：

- `start` 本身不处理数据，也不写变量。
- 它的任务只有一个，就是告诉系统“从哪里开始执行”。
- 通常会直接连到 `sequence`、`print`、`get_sensor_info` 或其他真正干活的节点上。

### 4.2 `print` 打印节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 输出调试信息或变量内容。 |
| 核心配置 | `messageSource`、`message`、`variableId`、`nextNodeId` |
| 数据行为 | 不负责生成业务结果，主要用于观察和调试。 |

详细说明：

- 当 `messageSource = manual` 时，打印固定文本。
- 当 `messageSource = variable` 时，打印某个工作流变量的值。
- 它非常适合放在关键节点后面，用来确认数据是否真的被写入变量。

适合使用它的地方：

- 确认数据库查询是否返回了结果。
- 确认分析摘要节点生成的文本长什么样。
- 调试某个分支到底走没走进去。

### 4.3 `sequence` 顺序节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 作为流程中的中间串联节点，也可以写备注。 |
| 核心配置 | `comment`、`nextNodeId` |
| 数据行为 | 不做数据处理，主要承担“组织流程”的职责。 |

详细说明：

- 这个节点本质上是一个“流程占位节点”。
- 它特别适合把一大段流程拆成几个阶段，例如“采集阶段”“分析阶段”“输出阶段”。
- 在复杂演示项目里，适当增加 `sequence` 节点会让流程图更容易看懂。

### 4.4 `loop` 循环节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 让一组节点重复执行。 |
| 核心配置 | `loopConditionType`、`loopCount`、`loopConditionExpr`、`bodyNodeIds`、`nextNodeId` |
| 数据行为 | 会多次执行循环体内的节点。 |

详细说明：

- 当前实现里，最稳定、最直接的是“按次数循环”。
- `loopConditionType = count` 时，使用 `loopCount` 控制循环次数。
- `loopConditionType = expr` 是预留的表达式模式，更多是结构上的保留能力。
- 循环体里的节点通过 `bodyNodeIds` 组织。
- 循环结束后，再通过 `nextNodeId` 进入下一段流程。

适合场景：

- 批量轮询多个采集动作。
- 多次测试某段逻辑是否稳定。
- 演示“周期采样”这种概念性流程。

### 4.5 `branch` 分支节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 根据条件走真分支或假分支。 |
| 核心配置 | `branchCondition`、`trueBodyNodeIds`、`falseBodyNodeIds`、`nextNodeId` |
| 数据行为 | 控制流程进入不同处理路径。 |

详细说明：

- 当前配置里，`branchCondition` 更像一个占位用的布尔结果。
- 真分支和假分支分别拥有自己的节点列表。
- 分支结束后可以再汇合到统一的后续节点。

适合场景：

- 判断设备是否异常。
- 判断是否触发告警。
- 判断是否进入“决策建议”或“普通展示”路径。

## 5. 数据处理与输出节点

这类节点是工作流和大屏联动的核心。

### 5.1 `get_sensor_info` 获取传感器信息

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 读取设备列表或最近传感器数据，并写入变量。 |
| 核心配置 | `source`、`deviceId`、`limit`、`targetVariableId`、`nextNodeId` |
| 默认行为 | `source` 默认是 `list_sensors`，`limit` 默认是 5。 |

详细说明：

- `source = list_sensors` 表示读取设备列表或传感器列表。
- `source = latest_data` 表示读取最近一次或最近几次传感器数据。
- `deviceId` 可选；留空时使用默认设备。
- `limit` 控制读取条数。
- 最终结果写入 `targetVariableId` 对应的变量。

变量类型差异：

| 目标变量类型 | 结果写法 |
| --- | --- |
| `string` | 保存 JSON 文本 |
| `int` | 保存记录条数 |

适合绑定的大屏组件：

- `agri-sensor`
- `text`
- `chart-bar`
- `chart-line`
- `chart-pie`

注意事项：

- 如果你希望后面接图表，最好把它整理成 CSV 字符串或让后续节点产出更适合图表的结构。
- 如果只是想直接显示设备/传感器信息，字符串变量会更实用。

### 5.2 `db_query` 数据库查询

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 执行只读 SQL 查询，并把结果写入变量。 |
| 核心配置 | `sql`、`targetVariableId`、`nextNodeId` |
| 限制 | 只支持只读查询，也就是 `SELECT`。 |

详细说明：

- 这个节点非常适合拿历史表、统计表或分析结果表做展示。
- 默认示例 SQL 就是按时间倒序取最近的温湿度记录。
- 返回结果同样会按照目标变量类型分别处理。

变量类型差异：

| 目标变量类型 | 结果写法 |
| --- | --- |
| `string` | 保存查询结果 JSON 文本 |
| `int` | 保存查询条数 |

适合绑定的大屏组件：

- `text`
- `chart-bar`
- `chart-line`
- `chart-pie`

注意事项：

- 如果你要绑定图表组件，工作流端口最好输出 CSV 字符串，而不是原始 JSON。
- 如果只是想看查询有没有结果，用整型变量统计条数就很方便。

### 5.3 `analytics_summary` 农业分析摘要

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 生成面向农业分析的摘要文本或摘要结构。 |
| 核心配置 | `analysisType`、`deviceId`、`hours`、`limit`、`targetVariableId` |
| 默认行为 | 默认任务是 `overview`，默认时间窗口 48 小时，默认条数上限 8。 |

这个节点的定位非常重要：

- 它不是原始数据采集节点。
- 它也不是深度结构化模型节点。
- 它更像“把分析结果整理成一段可读摘要”的节点。

支持的分析任务如下：

| 分析任务值 | 中文含义 | 更适合的展示目标 |
| --- | --- | --- |
| `overview` | 总览概况 | 首页总览、总述文字 |
| `timeline` | 趋势时序 | 趋势摘要、变化结论 |
| `alerts` | 风险告警 | 告警栏、异常摘要 |
| `recommendations` | 决策建议 | 建议列表、运营提示 |
| `forecast` | 气候预测 | 气候趋势摘要 |
| `yield` | 产量预测 | 产量结论摘要 |
| `decision` | 辅助决策 | 决策建议摘要 |
| `report` | 报告摘要 | 汇报页、报告型展示 |

为什么它不适合普通文本组件直接显示：

- 它的输出里可能带 JSON 外壳。
- 可能包含英文键名。
- 可能带多余的 `{}`、`[]`、`【】` 之类包裹符号。

因此，最佳搭配通常是：

```text
analytics_summary -> 字符串变量 -> output -> agri-summary
```

而不是直接：

```text
analytics_summary -> text
```

后者虽然能显示，但展示效果通常会比较“原始”。

### 5.4 `abstract_data_model` 农业环境抽象模型

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 基于历史传感器数据构建统一的农业环境抽象模型。 |
| 核心配置 | `deviceId`、`hours`、`minPoints`、`targetVariableId` |
| 默认行为 | 默认分析 168 小时数据，默认最少样本点为 24。 |

这个节点可以理解为当前项目里最“高级”、最适合大屏专题展示的农业分析节点。

它的特点是：

- 不只给一段摘要文字
- 还会输出更结构化的农业分析结果
- 结果中会附带适合大屏消费的 `screen_contract`

这个 `screen_contract` 很关键，因为它相当于帮大屏组件准备好了统一的取数协议。

它通常会涵盖这些方向：

- 环境概况
- 风险分数
- 主导维度
- 气候趋势
- 产量预测
- 决策建议

最佳搭配通常是：

```text
abstract_data_model
  -> 字符串变量
  -> output
  -> agri-model / agri-climate / agri-yield / agri-decision
```

适合它的原因是：

- `agri-model` 读模型概览
- `agri-climate` 读气候预测
- `agri-yield` 读产量信息
- `agri-decision` 读决策模块

### 5.5 `output` 输出端口节点

| 项目 | 说明 |
| --- | --- |
| 节点作用 | 把工作流变量暴露为项目端口。 |
| 核心配置 | `variableId`、`nextNodeId` |
| 系统角色 | 工作流与大屏应用之间的数据桥梁。 |

详细说明：

- 前面几个节点是在“生产结果”。
- `output` 节点是在“对外开放结果”。
- 大屏编辑器里能看到的端口，本质上来自这里。

这个节点是最容易被忽略，但实际上最关键的节点之一。

常见误区：

- 误区 1：变量已经有值了，大屏一定能看到。
- 实际情况：不一定，除非它经过了 `output`。

## 6. 大屏组件详解

大屏组件可以按“数据来源方式”分成两种体系。

### 6.1 第一类：支持绑定工作流端口的组件

这类组件的特点是：

- 可以手动输入内容
- 也可以绑定某个工作流项目的端口

当前属于这一类的组件有：

| 组件 | 是否支持工作流端口 | 支持的数据类型 |
| --- | --- | --- |
| `text` | 是 | `string`、`int` |
| `image` | 是 | `string` |
| 图表组件 | 是 | `string`、`csv` |
| `agri-model` | 是 | `string`、`csv` |
| `agri-climate` | 是 | `string`、`csv` |
| `agri-yield` | 是 | `string`、`csv` |
| `agri-decision` | 是 | `string`、`csv` |
| `agri-summary` | 是 | `string` |

这里有一个实现层面的细节：

- 左侧组件库展示的是 `chart-bar`、`chart-line`、`chart-pie`
- 但组件实际创建后，内部统一是 `chart` 类型，只是通过 `chartType` 区分柱状图、折线图和饼图

所以从使用角度看它们是三种图表，从实现角度看它们共享同一套图表组件能力。

### 6.2 第二类：主要依赖手动内容或接口轮询的组件

这类组件不走工作流端口绑定，而是用手动模拟或接口轮询。

| 组件 | 数据模式 | 说明 |
| --- | --- | --- |
| `agri-sensor` | 手动模拟 / 后端接口 | 主要展示传感器卡片列表。 |
| `weather` | 手动模拟 / 外部 API | 主要展示天气卡片。 |

也就是说：

- `agri-sensor` 更像“实时数据卡片”
- `weather` 更像“外部服务卡片”

而不是工作流分析结果卡片。

## 7. 各类大屏组件的作用和适用场景

### 7.1 `text` 文本展示

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 显示纯文字或简单数值。 |
| 数据来源 | 手动输入 / 工作流端口 |
| 适合内容 | 标题、说明、告警数字、简短结论 |

适合它的情况：

- 显示一个数值
- 显示一段短提示
- 显示整型统计结果

不适合它的情况：

- 直接显示复杂 JSON 摘要
- 显示结构化农业分析结果

### 7.2 `image` 图片展示

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 显示图片地址或上传图片。 |
| 数据来源 | 手动输入 / 工作流端口 |
| 端口要求 | 只能接字符串类型 |

适合场景：

- 摄像头抓拍图片
- 静态示意图
- 用字符串端口返回图片 URL 或 Base64

### 7.3 `chart-bar` / `chart-line` / `chart-pie` 图表组件

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 读取 CSV 数据并绘制图表。 |
| 数据来源 | 手动 CSV / 工作流端口 |
| 端口要求 | `string` 或 `csv` |

图表组件的核心不是“能显示任何数据”，而是“能显示符合图表结构的数据”。

这意味着：

- 如果工作流输出的是 JSON 字符串，图表通常不能直接画出来。
- 如果工作流输出的是 CSV 字符串，图表会更容易绑定成功。

图表支持的特性包括：

- 单系列、双系列、多系列
- 宽表布局
- 长表布局
- 超出容量后的自动聚合

适合场景：

- 温湿度时间曲线
- 多设备数据对比
- 各类告警数量分布
- 产量因子评分对比

### 7.4 `agri-sensor` 传感器数据

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 以卡片方式展示温度、湿度、光照、土壤湿度等数据。 |
| 数据来源 | 手动模拟 / 后端接口 |
| 适合内容 | 传感器实时值、状态灯、现场监控概况 |

这个组件更偏“实时监测面板”，不是分析结果面板。

适合搭在大屏首页，用来回答：

- 现在温室里的温度是多少
- 当前湿度是否正常
- 光照是否异常

### 7.5 `agri-model` 环境抽象模型

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 展示环境抽象模型、主导维度、风险分数和摘要。 |
| 数据来源 | 手动 JSON / 工作流端口 |
| 最佳绑定 | `abstract_data_model` 输出 |

这是一个“模型概览组件”。

它更适合回答：

- 当前环境整体状态如何
- 哪个维度是主导因素
- 风险值高不高

### 7.6 `agri-climate` 气候预测卡

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 展示未来气候趋势、微气候状态和关键环境预测。 |
| 数据来源 | 手动 JSON / 工作流端口 |
| 最佳绑定 | `abstract_data_model`、`forecast` 或 `climate` 类输出 |

它更偏“趋势与预测”视角，而不是静态总览视角。

### 7.7 `agri-yield` 产量预测卡

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 展示产量指数、亩产估算和影响因子评分。 |
| 数据来源 | 手动 JSON / 工作流端口 |
| 最佳绑定 | `abstract_data_model` 或产量预测类输出 |

它适合展示：

- 当前预测亩产
- 产量指数
- 哪些因素拉高或拉低了产量预测

### 7.8 `agri-decision` 辅助决策卡

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 展示优先动作、原因说明和模块化建议列表。 |
| 数据来源 | 手动 JSON / 工作流端口 |
| 最佳绑定 | `abstract_data_model` 或 `decision` 类输出 |

这个组件很适合放在大屏的“结论区”或“建议区”。

它主要回答：

- 接下来最应该做什么
- 为什么这么做
- 还有哪些次级建议

### 7.9 `agri-summary` 分析摘要卡

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 专门解析农业分析摘要文本，去掉原始 JSON 感。 |
| 数据来源 | 手动文本 / 工作流端口 |
| 端口要求 | 只能接字符串 |
| 最佳绑定 | `analytics_summary` 输出 |

这是当前项目里非常有辨识度的一个专用组件。

它的核心价值在于：

- 自动解析 JSON 文本
- 去掉多余的花括号、方括号、书名号
- 尽量剥离英文键名
- 把摘要整理成更适合阅读的结论、指标和建议

所以，当你遇到这种情况时：

“分析结果已经出来了，但普通文本组件显示出来很丑、很乱、很像原始接口返回值”

优先应该想到 `agri-summary`。

### 7.10 `weather` 天气信息

| 项目 | 说明 |
| --- | --- |
| 主要作用 | 展示外部天气信息。 |
| 数据来源 | 手动模拟 / 外部 API |
| 适合内容 | 温度、湿度、风速、天气描述 |

这个组件和农业工作流不是强绑定关系。

它更像是一个“外部环境补充组件”，适合和农业分析结果放在同一块大屏里形成上下文对照。

## 8. 数据绑定方式详解

### 8.1 哪些组件需要先保存工作流项目

当组件切到“工作流端口”模式时，它需要先选择“工作流项目”和“项目端口”。

这意味着：

- 工作流项目必须先保存
- 工作流里必须存在可用端口
- 端口的数据类型必须与组件兼容

如果某个组件切到工作流模式后显示“暂无可用端口”，常见原因有三种：

1. 工作流项目还没保存
2. 没有 `output` 节点
3. 有端口，但数据类型不匹配

### 8.2 数据类型兼容关系

| 组件 | 接收的数据类型 |
| --- | --- |
| `text` | `string`、`int` |
| `image` | `string` |
| 图表组件 | `string`、`csv` |
| `agri-model` | `string`、`csv` |
| `agri-climate` | `string`、`csv` |
| `agri-yield` | `string`、`csv` |
| `agri-decision` | `string`、`csv` |
| `agri-summary` | `string` |

### 8.3 手动模式和工作流模式的区别

| 模式 | 适用组件 | 含义 |
| --- | --- | --- |
| 手动输入 | 文本、图片、图表、农业分析类组件 | 直接在组件属性里填写内容进行预览和设计。 |
| 工作流端口 | 文本、图片、图表、农业分析类组件 | 运行时从工作流项目端口取值。 |
| 后端接口 | `agri-sensor` | 按刷新间隔轮询项目后端接口。 |
| 外部 API | `weather` | 按刷新间隔请求天气接口。 |

## 9. 推荐搭配关系

### 9.1 从“节点职责”角度看

| 工作流节点 | 最适合输出给谁 |
| --- | --- |
| `get_sensor_info` | `agri-sensor`、`text`、图表组件 |
| `db_query` | `text`、图表组件 |
| `analytics_summary` | `agri-summary` |
| `abstract_data_model` | `agri-model`、`agri-climate`、`agri-yield`、`agri-decision` |
| `output` | 所有需要绑定工作流数据的大屏组件 |

### 9.2 从“展示目标”角度看

| 你想展示什么 | 更适合的工作流节点 | 更适合的大屏组件 |
| --- | --- | --- |
| 实时环境现状 | `get_sensor_info` | `agri-sensor`、`text` |
| 历史变化趋势 | `db_query` 或可转成表格数据的节点 | `chart-line`、`chart-bar` |
| 农业分析总述 | `analytics_summary` | `agri-summary` |
| 模型总览 | `abstract_data_model` | `agri-model` |
| 气候预测 | `abstract_data_model` | `agri-climate` |
| 产量预测 | `abstract_data_model` | `agri-yield` |
| 决策建议 | `abstract_data_model` 或 `analytics_summary(decision)` | `agri-decision`、`agri-summary` |

## 10. 三种典型演示搭建思路

### 10.1 实时监测型大屏

推荐链路：

```text
start
  -> get_sensor_info
  -> output
  -> agri-sensor / text / chart-line
```

特点：

- 重点看实时值
- 重点看设备状态
- 适合首页概况

### 10.2 分析汇总型大屏

推荐链路：

```text
start
  -> analytics_summary
  -> output
  -> agri-summary
```

特点：

- 页面信息密度高
- 更适合做汇报展示
- 可以突出“系统会自动分析并输出结论”

### 10.3 智能决策型大屏

推荐链路：

```text
start
  -> abstract_data_model
  -> output
  -> agri-model / agri-climate / agri-yield / agri-decision
```

特点：

- 最能体现项目“智能农业分析”的特色
- 展示结果更结构化
- 适合放在演示项目的核心页面

## 11. 最容易踩的坑

### 11.1 用普通文本组件直接显示摘要节点结果

现象：

- 页面能显示
- 但会带英文键名、原始 JSON、括号和杂乱结构

原因：

- `text` 组件只是简单显示文本
- 它不负责解析农业摘要结构

正确做法：

优先改用 `agri-summary`。

### 11.2 工作流里明明有变量，大屏里却看不到

原因通常是：

- 没有 `output`
- 工作流项目没有保存
- 端口类型不匹配

### 11.3 图表绑定后没有图

常见原因：

- 传过去的是 JSON，不是 CSV
- 标签列和数值列没有配置好
- 数据列不是数值列

### 11.4 以为 `agri-sensor` 能直接绑定工作流端口

当前不是这种设计。

`agri-sensor` 更偏：

- 手动模拟展示
- 后端接口轮询展示

如果你想展示“工作流分析结果”，应优先用：

- `agri-summary`
- `agri-model`
- `agri-climate`
- `agri-yield`
- `agri-decision`

### 11.5 以为 `weather` 属于工作流分析链路

也不是。

`weather` 更偏外部环境补充信息，和农业工作流是并列关系，不是上下游关系。

## 13. 最推荐的理解方式

如果把整套系统压缩成一句话，可以这样理解：

`工作流节点负责产出数据，工作流变量负责暂存数据，output 节点负责公开数据，大屏组件负责把数据变成可视化结果。`

如果把项目特色再提炼一步，可以这样理解：

- `analytics_summary` 代表“把复杂分析结果整理成可读摘要”
- `agri-summary` 代表“把摘要进一步变成更适合大屏阅读的内容”
- `abstract_data_model` 代表“把农业分析结果结构化成统一模型”
- `agri-model / climate / yield / decision` 代表“把统一模型拆成不同的专题卡片展示”

## 14. 新增：`advanced_prediction` 高级预测与可视化节点

这个节点用于把“高级预测分析”直接接入工作流，并把结果拆成更适合大屏绑定的两类输出：

- 图片 URL：适合直接绑定到大屏图片组件
- CSV 文本：适合直接绑定到大屏图表组件

### 14.1 它解决了什么问题

以前工作流里的分析结果大多偏 JSON 或摘要文本，不适合直接拿去做图片展示或图表展示。

这个节点补上的能力是：

- 在工作流里直接生成预测曲线图、原始趋势图、热力图、异常图等图片地址
- 在工作流里直接生成模型对比 CSV、预测序列 CSV
- 再通过 `output` 节点暴露给大屏应用

### 14.2 推荐输出内容

| 输出内容 | 推荐变量类型 | 推荐绑定组件 |
| --- | --- | --- |
| `forecast_plot_url` | `string` | `image` |
| `raw_plot_url` | `string` | `image` |
| `feature_importance_plot_url` | `string` | `image` |
| `corr_heatmap_plot_url` | `string` | `image` |
| `anomaly_plot_url` | `string` | `image` |
| `model_compare_csv` | `csv` | `chart-bar`、`chart-line` |
| `forecast_series_csv` | `csv` | `chart-line` |
| `decision_report_json` | `string` | `text`、`agri-summary` |
| `full_result_json` | `string` | 结构化组件或后续扩展 |

### 14.3 最推荐的用法

如果你要在大屏里显示预测图：

```text
start
  -> advanced_prediction（输出 forecast_plot_url）
  -> output
  -> image
```

如果你要在大屏里显示模型对比图表：

```text
start
  -> advanced_prediction（输出 model_compare_csv）
  -> output
  -> chart-bar / chart-line
```

### 14.4 使用注意事项

- 当输出是图片 URL 时，变量建议设置成 `string`。
- 当输出是图表数据时，变量建议设置成 `csv`。
- 大屏图片组件只接受字符串端口，所以不能把 JSON 直接绑定给图片组件。
- 大屏图表组件更适合绑定 CSV 文本，不适合直接绑定原始 JSON。
- 这个节点支持“优先重新生成”和“优先复用已有结果”两种策略，适合分别应对实时演示和快速回放两种场景。
